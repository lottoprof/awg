# AGENTS.md

## Основной файл инструкций
Этот файл — точка входа для агента в данном репозитории.
При любой работе с репозиторием агент сверяется с этим файлом первым.

## Назначение репозитория
Репозиторий содержит систему развертывания `AmneziaWG` на чистых `Debian 12 VPS`:
- `docs/` — архитектурные и операционные документы;
- `script_srv/` — серверные backend-скрипты;
- `script_client/` — локальные операторские скрипты;
- `templates/` — шаблоны конфигов;
- `awg_clients/` — локальный каталог для полученных клиентских `.conf` и `.png`.
- `awg_localhost/` — локальные артефакты и документация по настройке `AWG` на рабочей машине.

## Стартовые документы агента
Перед выполнением любой задачи агент читает документы в следующем порядке:

1. `docs/AgentManifest.md` — единый входной манифест: параметры, defaults, правила интерпретации.
2. `docs/AgentExecutionRules.md` — правила выполнения: решения, идемпотентность, парсинг вывода.
3. `docs/DeploymentPlan.md` — план развертывания и порядок этапов.

Порядок ниже не задает строгую последовательность выполнения.
`docs/ServerLayout.md` и `docs/ServerConfigFormat.md` являются справочными документами и читаются до `docs/ServerConfigInit.md`.

## Приоритет источников
При конфликте между документами приоритет применяется в следующем порядке:
1. `docs/AgentManifest.md` — высший приоритет по параметрам и defaults.
2. `docs/AgentExecutionRules.md` — высший приоритет по правилам выполнения.
3. Операционные документы — по конкретным шагам и командам.

## Правила работы агента

### Общие
- Сначала планировать, потом выполнять действия.
- Если задача касается настройки, развертывания или эксплуатации сервера, сначала читать и синхронизировать документы, потом менять код или запускать шаги.
- Не додумывать значения параметров — использовать только то, что зафиксировано в манифесте или явно передано.
- Если обязательный параметр не заполнен и не имеет default, остановиться и запросить значение.
- Каждый критичный шаг выполнять по схеме `check -> apply -> verify`.
- Если проверка уже показывает целевое состояние, шаг не повторять.
- После изменения кода обязательно выполнять профильные проверки для соответствующего языка и инструмента: lint, type checks, syntax checks и tests, если они применимы.
- Для shell-скриптов перед завершением работы обязательно выполнять `bash -n` и `shellcheck`.

### Идемпотентность
- Перед созданием пользователя проверить его существование через `id <user>`.
- Перед записью файла конфигурации проверить его наличие и валидность.
- `iptables -A` не считать идемпотентной операцией.
- На новом сервере перед применением правил выполнять сброс через `iptables -F`.
- Подробные правила по каждому типу операции — в `docs/AgentExecutionRules.md`.

### Принятие решений
- Необходимость reboot определять по правилам из `docs/AgentExecutionRules.md`, не по косвенным признакам.
- Переход в manual build (`docs/AmneziaWGManualBuild.md`) допустим только после нормализации ядра и повторной попытки пакетной установки.
- При неоднозначном результате шага не продолжать, а запросить подтверждение.

### Граница автоматизации
Агент не автоматизирует:
- импорт клиентского конфига в мобильное приложение;
- включение туннеля на клиентском устройстве;
- проверку внешнего IP на клиентской стороне;
- изменение `~/.ssh/config` на рабочей машине оператора.

Эти шаги выполняет оператор вручную.
Агент продолжает работу после подтверждения.

## Скрипты
Контракт поведения серверных скриптов зафиксирован в `docs/ClientManagement.md`.
Контракт локального CLI зафиксирован в `docs/Awgctl.md`.

Реализация скриптов находится в:
- `script_srv/awg-add-client`
- `script_srv/awg-list-clients`
- `script_srv/awg-export-client`
- `script_srv/awg-revoke-client`
- `script_client/awgctl`

Указанные файлы уже являются текущей реализацией `v1`.

Расхождение между документом и реализацией считается дефектом.
При изменении поведения обновлять синхронно и документ, и скрипт.
После изменения shell-скриптов обязательна проверка:
- `bash -n`
- `shellcheck`

## Правила документации
- Все рабочие документы хранить в `docs/`.
- Новые документы писать по структуре `docs/DeploymentPlan.md`.
- Документы должны содержать: цель, входные условия, команды, проверки результата, ожидаемый итог.
- Стиль — без обращения к читателю, формулировки в виде действия: `Проверить`, `Установить`, `Создать`.
