# Правила выполнения для AI-агента

## Цель
Зафиксировать, как AI-агент должен принимать решения, повторно запускать шаги и разбирать вывод команд при развертывании `AWG`.

## Назначение
Этот документ дополняет [docs/AgentManifest.md](AgentManifest.md).

Разделение ролей:
- [docs/AgentManifest.md](AgentManifest.md) отвечает за входные параметры и defaults;
- этот документ отвечает за execution rules;
- operational docs отвечают за последовательность действий.

## Общие правила
- Каждый критичный шаг должен иметь `check -> apply -> verify`.
- Если проверка уже показывает целевое состояние, агент не должен повторно применять шаг.
- Если результат шага неоднозначен, агент не должен додумывать его по косвенным признакам.
- Если входной параметр не заполнен и не имеет default из манифеста, агент должен остановиться и запросить значение.
- Если в репозитории уже есть скрипт реализации, агент должен использовать его как основной исполняемый артефакт.

## Decision Rules
### Определение необходимости reboot после обновления ядра
Команды:
```bash
uname -r
apt-cache policy linux-image-amd64 linux-headers-amd64
```

Правило:
- если после `apt upgrade` или установки из backports был установлен новый kernel package, агент должен считать reboot обязательным;
- если `dkms status` показывает модуль только под ядро, отличное от `uname -r`, агент должен считать reboot обязательным;
- если модуль `amneziawg` собран под текущее `uname -r`, reboot по причине DKMS не требуется.

### Успешность пакетной установки `AmneziaWG`
Пакетный сценарий считается успешным только если одновременно выполнено все:
- `apt install amneziawg` завершился без ошибки;
- `which awg` и `which awg-quick` возвращают пути;
- `modprobe amneziawg` завершается без ошибки;
- `dkms status` не противоречит текущему `uname -r`.

### Когда переходить в manual build
Переход в [docs/AmneziaWGManualBuild.md](AmneziaWGManualBuild.md) допустим только после такого порядка:
1. Проверить `uname -r`.
2. Проверить наличие headers под текущее ядро.
3. При необходимости нормализовать ядро и headers.
4. Выполнить reboot.
5. Повторить пакетную установку.
6. Только после этого переходить в manual build.

### Успешность настройки `awg-quick`
Этап считается успешным только если одновременно выполнено все:
- `sudo awg-quick strip awg0` завершается без ошибки;
- `systemctl start awg-quick@awg0` или `restart` завершается без ошибки;
- `systemctl status awg-quick@awg0 --no-pager` показывает `active (running)` или эквивалентное активное состояние;
- `ip addr show awg0` показывает интерфейс.

### Успешность добавления клиента
`awg-add-client` считается успешным только если одновременно выполнено все:
- в `/etc/amnezia/amneziawg/awg0.conf` появился новый `[Peer]`;
- в `clients.tsv` появилась запись клиента;
- созданы `.conf` и `.png` в основном хранилище, если QR включен и доступен;
- созданы копии в `EXPORT_DIR`;
- `awg-quick@awg0` успешно перезапущен.

### Успешность smoke-test
Smoke-test считается успешным только если одновременно выполнено все:
- тестовый клиент создался без ошибки;
- `awg show awg0 latest-handshakes` показывает handshake;
- `awg show awg0 endpoints` показывает endpoint клиента;
- клиентский трафик проходит;
- внешний IP клиента совпадает с IP сервера.

Правило:
- создание клиента и серверные проверки может выполнять агент;
- импорт клиентского конфига в реальное клиентское устройство и включение туннеля считаются ручным действием пользователя;
- smoke-test в `v1` является частично автоматизируемым.

## Idempotency Rules
### Пакеты
- `apt install -y ...` считать идемпотентным шагом.
- Повторный запуск допустим без дополнительных проверок.

### Создание пользователя
Команды `useradd` + `chpasswd` для `<admin_user>` не считаются идемпотентными.

Правило:
- до выполнения проверить:
```bash
id <admin_user>
```
- если пользователь уже существует, шаг создания пропустить;
- шаг добавления в группу `sudo` выполнить отдельно.
- если пользователь создается впервые, агент может задать временный `admin_temp_password`;
- после завершения установки агент должен потребовать ручную смену этого пароля.

### Добавление в группу `sudo`
Команда:
```bash
id <admin_user>
```

Правило:
- если пользователь уже состоит в группе `sudo`, повторное `usermod -aG sudo` допустимо, но не обязательно;
- агент может пропустить этот шаг, если группа уже видна в `id`.

### Создание каталогов
Команды `mkdir -p` считаются идемпотентными.

Правило:
- повторный запуск допустим;
- после повторного запуска агент все равно должен проверить владельца и права.

### Выставление прав
Команды `chown` и `chmod` считаются идемпотентными.

Правило:
- повторный запуск допустим;
- после этого агент должен проверять фактические mode и owner.

### Создание `/etc/amnezia/params`
Создание файла не считается идемпотентным по умолчанию.

Правило:
- если файл уже существует, агент не должен безусловно перезаписывать его;
- сначала нужно проверить наличие обязательных ключей;
- если формат корректен и значения согласованы, шаг создания пропустить;
- если формат некорректен, агент должен остановиться и потребовать явного решения.

### Создание `/etc/amnezia/amneziawg/awg0.conf`
Создание файла не считается идемпотентным по умолчанию.

Правило:
- если файл уже существует, агент должен проверить наличие `[Interface]` и базовых параметров;
- если файл уже рабочий, агент не должен перезаписывать его;
- если файл невалиден, сначала сохранить backup, потом исправлять.

### Создание `clients.tsv`
Команда `touch` идемпотентна, но реестр не должен затираться.

Правило:
- если файл уже существует, агент не должен его очищать;
- повторный запуск может только проверить наличие и права.

### Добавление APT repository
Добавление repo-файла не считается безусловно идемпотентным.

Правило:
- если файл repo уже существует с тем же содержимым, шаг пропустить;
- если файл существует с другим содержимым, агент должен остановиться и запросить решение;
- `apt update` после проверки файла выполнять допустимо повторно.

### Firewall
Применение правил `iptables -A` не считается идемпотентным.

Правило:
- если документ требует полную новую схему на чистом сервере, агент может очищать ruleset перед применением только при явном условии, что сервер новый;
- если сервер не новый, агент не должен автоматически выполнять `iptables -F` и `-X`.

### `awg-add-client`
Не считается идемпотентным.

Правило:
- повторный запуск с тем же именем создает новый `client_id` только если это предусмотрено реализацией;
- агент не должен повторно вызывать `awg-add-client` после частичной ошибки без проверки, какие файлы и записи уже созданы.
- после частичной ошибки агент должен проверить:
  - появился ли новый `[Peer]` в `/etc/amnezia/amneziawg/awg0.conf`;
  - появилась ли запись в `clients.tsv`;
  - созданы ли `.conf` и `.png` в основном хранилище;
  - созданы ли копии в `EXPORT_DIR`;
  - поднялся ли `awg-quick@awg0` после попытки перезапуска.
- если `[Peer]` добавлен, но `clients.tsv` или artifacts не созданы, состояние считать частично примененным и требующим ручного решения или rollback через backup;
- если `awg0.conf` уже восстановлен из backup и запись в `clients.tsv` не появилась, попытку считать неуспешной и не повторять автоматически;
- если `awg-quick@awg0` не поднялся после добавления клиента, перейти к диагностике сервиса и не выполнять повторный `awg-add-client`.

### `awg-export-client`
Считается условно идемпотентным.

Правило:
- повторный запуск допустим;
- должен только заново копировать существующие client artifacts в `EXPORT_DIR`.

### `awg-revoke-client`
Не считается полностью идемпотентным.

Правило:
- перед выполнением проверить текущий статус клиента в `clients.tsv`;
- если статус уже `revoked`, повторный отзыв без `--purge` можно пропустить;
- если используется `--purge`, агент должен отдельно проверить существование файлов.

## Parsing Rules
### Определение внешнего интерфейса
Команда:
```bash
ip route get 1.1.1.1
```

Правило:
- взять значение после `dev`;
- пример:
```text
1.1.1.1 via 85.85.249.1 dev eth0 src 85.85.249.249
```
- в этом примере `server_iface=eth0`.

### Определение текущего ядра
Команда:
```bash
uname -r
```

Правило:
- использовать весь вывод строки как значение текущего kernel release;
- не обрезать суффиксы вроде `-amd64`.

### Проверка `dkms status`
Команда:
```bash
dkms status
```

Правило:
- агент должен найти запись `amneziawg/...`;
- если в записи kernel release совпадает с `uname -r`, сборку считать подходящей;
- если запись есть только для другого kernel release, считать, что нужен reboot или дополнительная установка под текущее ядро.

Пример подходящего состояния:
```text
amneziawg/1.0.0, 6.1.0-43-amd64, x86_64: installed
```

при:
```text
uname -r -> 6.1.0-43-amd64
```

### Проверка загрузки модуля
Команды:
```bash
sudo modprobe amneziawg
lsmod | grep amneziawg
```

Правило:
- `modprobe` без ошибки считать положительным результатом;
- `lsmod` должен показывать `amneziawg`.

### Проверка SSH-конфига
Команда:
```bash
sshd -t
```

Правило:
- пустой вывод и нулевой код возврата считать успехом;
- любой stderr или ненулевой код считать блокирующей ошибкой.

### Проверка статуса systemd unit
Команда:
```bash
systemctl status <unit> --no-pager
```

Правило:
- искать `Active: active (running)` как положительный результат;
- `failed`, `inactive`, `activating (auto-restart)` считать неуспехом;
- при сомнении агент должен дополнительно проверить код:
```bash
systemctl is-active <unit>
```

### Проверка handshake
Команды:
```bash
awg show awg0 latest-handshakes
awg show awg0 endpoints
```

Правило:
- наличие endpoint и ненулевого/актуального handshake считать положительным результатом;
- отсутствие endpoint или отсутствие handshake считать неуспехом;
- при сомнении дополнительно смотреть `transfer`.

### Проверка чтения `params`
Команда:
```bash
set -a
. /etc/amnezia/params
set +a
```

Правило:
- отсутствие shell error считать успехом;
- если `source` завершился ошибкой, `params` считать невалидным.

### Локальный SSH alias
Для `v1` агент не должен автоматически изменять `~/.ssh/config` на рабочей машине оператора.

Правило:
- `server_alias` должен быть уже создан оператором;
- если alias отсутствует, агент должен остановиться и запросить явный SSH host или ручное создание alias.
