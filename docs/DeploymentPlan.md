# План разработки системы развертывания AWG на чистых VPS

## Цель
Собрать повторяемую систему развертывания `awg` на чистом VPS-сервере с минимальным количеством ручных действий, предсказуемой проверкой результата и возможностью безопасного повторного запуска.

## Что считаем результатом
- Новый сервер поднимается с нуля по одной инструкции или одной команде.
- Сервер получает базовое усиление безопасности до публикации VPN.
- `AmneziaWG` устанавливается и запускается без ручной правки системных файлов на каждом хосте.
- Конфигурация сервера и клиентов генерируется из шаблонов.
- После развертывания есть понятная проверка: сервис поднят, порт слушается, handshake проходит, маршрутизация работает.

## Ограничения первой версии
- Поддерживаем один дистрибутив: `Debian 12`.
- Поддерживаем один тип инсталляции: один VPS = один сервер `awg`.
- IPv4 обязателен, IPv6 откладываем.
- Управление клиентами идет через серверные backend-команды и локальный `awgctl`.
- Секреты на первом этапе храним локально в защищенных файлах, без внешнего secret manager.

## Этап 1. Базовая спецификация
Определить неизменяемые входные параметры:
- IP сервера.
- SSH-порт.
- имя административного пользователя.
- сетевой интерфейс сервера.
- подсеть `awg`.
- порт `awg`.
- DNS для клиентов.
- набор параметров обфускации `Jc/Jmin/Jmax/S1/S2/H1-H4`.

Выход этапа:
- файл с перечнем обязательных переменных;
- шаблон `.env` или `.yaml` с примером значений;
- правило именования серверов, клиентов и конфигов.

## Этап 2. Архитектура развертывания
Выбран формат системы: `bash`.

Базовый принцип:
- один главный entrypoint;
- отдельные модули по этапам;
- общая библиотека функций;
- шаблоны конфигов вместо точечной правки файлов;
- поддержка повторного запуска без дублирования состояния.

Требования к shell-архитектуре:
- `set -euo pipefail`;
- единый логгер;
- разделение `check/apply/verify` для критичных этапов;
- явные переменные в `config.sh`;
- минимизация хрупких `sed`-правок, где можно генерировать файл целиком.

Выход этапа:
- список shell-модулей и их ответственности;
- схема: локальная машина оператора -> SSH -> VPS.

## Этап 3. Инициализация чистого сервера
Автоматизировать первичную подготовку:
- базовое обновление пакетов;
- установка `tmux`, `vim`, `curl`, `wget`, `sudo` и build toolchain;
- ранняя проверка kernel/headers;
- ранняя пакетная установка `AmneziaWG`;
- ранний reboot, если модуль собирается под новое ядро;
- только после этого создание административного пользователя;
- установка SSH-ключей;
- отключение входа по паролю;
- смена SSH-порта;
- запрет root login.

Выход этапа:
- скрипт `bootstrap`;
- отдельная проверка, что `AmneziaWG` и ядро приводятся в рабочее состояние еще до жесткого изменения SSH-доступа;
- отдельная проверка, что новый SSH-доступ работает до отключения старого режима.

## Этап 4. Сетевое и системное основание
Подготовить сервер к работе `awg`:
- включить `ip_forward`;
- настроить firewall;
- настроить `iptables` и NAT;
- установить зависимости сборки;
- при необходимости поставить kernel и headers из backports.

Выход этапа:
- шаблон сетевых правил;
- отдельный smoke-check после применения firewall.

## Этап 5. Установка AmneziaWG
Определить источник установки:
- community installer;
- собственная сборка из репозитория;
- готовые пакеты, если появятся стабильные.

Для первой версии практичнее идти так:
- сначала нормализовать kernel/headers;
- затем выполнить пакетную установку;
- ранний reboot делать сразу, если модуль собирается под новое ядро;
- manual build использовать только как fallback после неудачи пакетного пути на нормализованном ядре.

Выход этапа:
- автоматизированный install step;
- документ с зависимостями и точкой обновления версии.

## Этап 6. Генерация конфигурации сервера
Автоматизировать:
- генерацию ключей сервера;
- подготовку `awg0.conf` из шаблона;
- запись параметров сети и обфускации;
- включение и запуск `awg-quick@awg0`.

Выход этапа:
- шаблон серверного конфига;
- каталог для секретов;
- правило, какие данные генерируются один раз, а какие можно пересоздавать.

## Этап 7. Генерация клиентских конфигов
Подготовить отдельный поток для клиентов:
- генерация `client_private`, `client_public`, `psk`;
- добавление peer в серверный конфиг;
- выпуск клиентского `.conf`;
- опционально генерация QR.

Выход этапа:
- серверные backend-команды `add-client`, `list-clients`, `export-client`, `revoke-client`;
- каталог клиентов;
- таблица соответствия `client name -> IP -> public key`.

## Этап 8. Операторский workflow
После настройки сервера оператор должен работать со своего рабочего места, а не через постоянный интерактивный SSH-shell.

Нужно предусмотреть:
- локальную команду верхнего уровня;
- вызов серверных `awg-*` команд через `ssh`;
- получение `.conf` и `.png` на рабочее место оператора;
- очистку транзитной серверной зоны после успешной передачи.

Выход этапа:
- документ по операторскому workflow;
- спецификация локального CLI;
- схема `operator workstation -> ssh/rsync -> server backend`.

## Этап 9. Проверка и приемка
После каждого развертывания проверяем:
- SSH доступ по новому пользователю и порту;
- активность `awg-quick@awg0`;
- наличие интерфейса `awg0`;
- прослушивание UDP-порта;
- правила NAT/forward;
- внешний handshake тестовым клиентом;
- выход клиента в интернет через туннель.

Выход этапа:
- чек-лист проверки;
- автоматический `smoke.sh` или `verify.sh`.

## Этап 10. Эксплуатация
Нужно предусмотреть:
- добавление клиента;
- отзыв клиента;
- ротацию ключей;
- обновление ядра и `awg`;
- резервное копирование конфигов и секретов;
- восстановление сервера после пересоздания VPS.

Выход этапа:
- набор операционных процедур;
- инструкции на типовые операции day-2.

## Этап 11. Безопасность и контроль качества
До перехода к продакшен-использованию проверить:
- где хранятся приватные ключи;
- кто имеет доступ к конфигу и секретам;
- есть ли защита от случайной блокировки SSH;
- есть ли dry-run режим;
- можно ли повторно выполнять развертывание без поломки состояния.

Выход этапа:
- минимальный security checklist;
- минимальный rollback plan.
