# Работа оператора с рабочего места

## Цель этапа
Зафиксировать модель эксплуатации, в которой оператор не работает через постоянный интерактивный вход на сервер, а управляет `AWG` со своего рабочего места.

## Принцип
Оператор работает локально.

Серверные команды:
- `awg-add-client`
- `awg-list-clients`
- `awg-export-client`
- `awg-revoke-client`

не являются основным пользовательским интерфейсом оператора.

Они считаются backend-командами на сервере.

Основной интерфейс оператора должен быть локальным CLI, который:
- подключается к серверу по `ssh`;
- вызывает нужную серверную команду;
- при необходимости забирает файлы из `/home/awgesrv/export`;
- очищает транзитную зону после успешной передачи.

Модель прав:
- `list`, `add`, `export`, `revoke` должны работать через точечные `sudoers`-разрешения.

## Что не считаем правильным процессом
Не считаем целевым такой сценарий:
```bash
ssh awgesrv
sudo awg-add-client test
sudo awg-list-clients
sudo awg-export-client client01_test
sudo awg-revoke-client client01_test
```

Это допустимо для отладки и ручной диагностики, но не как постоянная схема работы оператора.

## Что считаем правильным процессом
Оператор должен работать со своего рабочего места командами уровня orchestration.

Целевой вид:
```bash
awgctl list-clients
awgctl add-client test
awgctl export-client client01_test
awgctl revoke-client client01_test
```

Локальная команда `awgctl` должна:
- знать SSH-alias сервера;
- вызывать на сервере нужный `awg-*` скрипт;
- забирать результат на локальную машину;
- минимизировать ручные действия оператора.

## Разделение ответственности
### Сервер
На сервере остаются:
- реальный `awg0.conf`;
- client registry;
- generated client configs;
- QR;
- backend-команды `awg-*`.

### Рабочее место оператора
На рабочем месте должны появиться:
- локальная CLI-обертка в `script_client/`;
- локальный каталог `awg_clients/` для полученных клиентских файлов;
- SSH-конфиг/alias для доступа к серверу;
- при необходимости `rsync` для забора файлов из транзитной зоны.

## Роль `/home/awgesrv/export`
`/home/awgesrv/export` остается транзитной зоной на сервере.

Правила:
- файлы попадают туда только для передачи оператору;
- держать их там постоянно не нужно;
- после успешного получения на рабочее место оператор должен очищать `export`;
- повторная выдача должна выполняться через `awg-export-client`.

## Минимальный целевой набор локальных операций
Оператор должен уметь со своего рабочего места:
- посмотреть список клиентов;
- создать клиента;
- получить `.conf` и `.png` локально;
- отозвать клиента;
- повторно выгрузить клиентский конфиг;
- очистить серверную транзитную зону после успешного получения.

## Действия оператора
Фактические действия оператора должны быть такими:
1. Выполнить `awgctl list-clients` и проверить текущий список.
2. При создании нового клиента выполнить `awgctl add-client <name>`.
3. Забрать полученные файлы из `awg_clients/` и передать их пользователю.
4. Если конфиг нужен повторно, выполнить `awgctl export-client <client_id>`.
5. Если клиента нужно отключить, выполнить `awgctl revoke-client <client_id>`.

## Что это меняет в проекте
Следующий слой разработки должен быть не серверным, а операторским.

То есть после набора серверных `awg-*` команд нужно проектировать:
- локальный `awgctl`;
- правила передачи файлов с сервера на рабочее место;
- правила очистки `/home/awgesrv/export`;
- минимальные требования к окружению рабочего места оператора.

## Проверки результата этапа
- Зафиксировано, что оператор работает локально, а не через постоянный SSH-shell.
- Зафиксировано, что `awg-*` на сервере являются backend-командами.
- Зафиксировано, что `/home/awgesrv/export` является транзитной зоной.
- Зафиксирована необходимость локального CLI поверх серверных команд.

## Ожидаемый итог

- не как набор серверных shell-команд для ручного SSH;
- а как операторский workflow с рабочего места.
